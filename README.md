# 사이드 프로젝트 작업 구조

여러 아이디어를 목업으로 찍어보고, 마음에 드는 것만 발전시키고, 나머지는 기록으로 남기기 위한 최소한의 관리 구조.

## 상태(status) 4단계

| 상태 | 의미 |
|---|---|
| 아이디어 | 아직 목업도 없음. 보통 허브의 "아이디어 메모"에만 적어두면 충분, 폴더까지 만들 필요 없음 |
| 프로토타입 | 목업을 만들어서 써보는 중. 계속할지 아직 결정 전 |
| 발전중 | 마음에 들어서 계속 키우는 중 (기능 추가, 실사용, 배포 준비 등) |
| 히스토리 | 써봤지만 계속 안 하기로 함. 파일은 지우지 않고 그대로 두고, 허브에는 접힌 기록으로만 표시 |

## 단일 소스: `registry.js`

전체 프로젝트 목록과 아이디어 백로그는 **[registry.js](registry.js) 한 곳에만** 적는다. [허브(index.html)](index.html)는 이 파일을 읽어서 카드를 그리므로, 허브 HTML을 직접 손대지 않아도 된다.

`index.html`이 저장소 **루트**에 있는 이유: GitHub Pages/Netlify/Vercel 같은 정적 호스팅은 루트의 `index.html`을 홈페이지로 서빙한다. 허브의 React 소스(옛 버전 포함)는 [작업실 허브/](작업실%20허브/) 폴더에 참고용으로만 남아 있다.

```js
window.REGISTRY = {
  statuses: [...],   // 상태별 라벨/색 정의 (건드릴 일 거의 없음)
  projects: [ { id, title, folder, file, date, status, tags, description, note }, ... ],
  ideas: [ "아이디어 문장", ... ]
};
```

## 새 아이디어가 생기면

1. **아직 목업 안 만들 거면** → 허브 페이지 하단 "아이디어 메모"에 바로 입력 (이 기기에 저장됨). 나중에 손이 가면 `registry.js`의 `ideas` 배열에도 옮겨 적어두면 다음에 새 기기/브라우저에서도 보임.
2. **목업을 만들기로 하면** → 폴더 하나 새로 생성 (`{프로젝트명}/`), 그 안에 `{slug}.jsx`(소스) + `{slug}.html`(더블클릭으로 바로 열리는 단일 파일 버전)를 넣는다. 설계 메모가 있으면 `{slug}-architecture.md` 같은 이름으로 같이 둔다.
3. `registry.js`의 `projects` 배열에 항목 하나 추가, `status: "프로토타입"`으로 시작.

## 써보고 난 뒤

- **마음에 들면** → `registry.js`에서 그 프로젝트의 `status`를 `"발전중"`으로만 바꾼다. 폴더/파일 이름은 그대로 둬도 되고, 기능이 크게 늘면 그때 새 폴더로 옮겨도 된다 (체크노트가 이 케이스: 원래 프로토타입은 `체크노트/`에 남겨두고, localStorage 저장을 붙인 실사용 버전은 `체크노트_실사용/`에 따로 둠).
- **접기로 하면** → `status`를 `"히스토리"`로 바꾸기만 하면 된다. 폴더/파일은 지우지 않는다 — 허브의 "히스토리" 섹션에 자동으로 접혀서 들어간다.

## 파일 규칙

- 폴더명: 한글 프로젝트명 그대로 (`아기랑 갈곳`, `초간단 배당현황` 등)
- 파일명: 영문 slug (`baby-place-registry`, `dividend-passbook` 등) — 폴더명과 문자 그대로 안 맞아도 됨
- `.jsx`: React 소스 (나중에 Vite 등으로 옮길 때 대비한 원본)
- `.html`: 실제로 열어보는 단일 파일 버전. **이게 "게시본"** — 허브 카드는 이 파일로 링크한다

## 왜 이렇게 했는가

기존에는 허브 파일이 두 벌(`maker-journal.jsx`, `maker-journal-hero.jsx`) 있었고 프로젝트 데이터를 양쪽에 손으로 따로 적다 보니 벌써 어긋나 있었다(같은 프로젝트를 다른 이름으로 부르거나, 실제로는 5개인데 2개만 반영). `registry.js` 하나로 합쳐서 "고칠 곳은 항상 한 곳"으로 만들고, 접었다/키운다는 결정을 상태값 하나 바꾸는 걸로 끝나게 했다.

## 배포 워크플로우 (git → live)

1. 로컬에서 수정 (파일 편집 / `registry.js` 상태 변경 등)
2. **승인 후에만** `git add` + `git commit` + `git push` — push되는 순간 연결된 호스팅에 그대로 반영됨
3. 즉, "커밋해서 push하기 전"까지는 로컬에만 있는 초안이고, push해야 실제 라이브 사이트에 반영된다

승인 없이 임의로 push하지 않는다. 로컬 커밋까지는 자유롭게 쌓아도 되지만, 원격(`git push`)은 매번 명시적으로 확인받고 진행한다.

저장소/호스팅 연결(최초 1회, 계정이 필요한 부분이라 직접 해야 함):
1. GitHub에 빈 저장소 하나 생성
2. 이 폴더에서 `git remote add origin <저장소 URL>` 후 첫 push
3. 호스팅 서비스(GitHub Pages / Netlify / Vercel 중 택1)에서 그 저장소를 "루트 디렉터리, 빌드 명령 없음(정적 파일 그대로 서빙)"으로 연결
4. 이후로는 `main` 브랜치에 push될 때마다 자동으로 라이브에 반영됨
